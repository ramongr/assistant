# RBS generator

`bin/assistant-rbs` (installed as the `assistant-rbs` executable when
the gem is added to a `Gemfile`) scans a source tree for
`Assistant::Service` subclasses and emits per-class RBS sidecar
signatures into a target directory. See the
[Using `bin/assistant-rbs` for Steep users](../guides/inputs.md#using-assistant-rbs-for-steep-users)
section of the Inputs guide for the design rationale (and the R1
limitation it works around).

Given a service like:

```ruby
# lib/services/create_user.rb
class CreateUser < Assistant::Service
  input :email, type: String, required: true
  input :name,  type: String, required: true
  input :role,  type: [String, Symbol], optional: true

  def execute
    { email:, name:, role: role || :member }
  end
end
```

run:

```text
$ bundle exec assistant-rbs lib --output sig
sig/services/create_user.rbs (generated)

$ cat sig/services/create_user.rbs
class CreateUser < Assistant::Service
  def email: () -> String
  def email?: () -> bool
  def valid_type_email?: () -> bool
  def valid_required_email?: (?bool log) -> bool

  def name: () -> String
  def name?: () -> bool
  def valid_type_name?: () -> bool
  def valid_required_name?: (?bool log) -> bool

  def role: () -> (String | Symbol | nil)
  def role?: () -> bool
  def valid_type_role?: () -> bool
end
```

Steep can now type-check call sites that read `CreateUser.new(...).email`
even though the getter is defined via `define_method` at class-load
time.

Notes:

* `--output sig` writes alongside any existing `sig/` tree; the
  generator only owns files it tagged with its `# generated by
  assistant-rbs` marker — hand-written sidecars are left untouched.
* Re-run after each `Service.input` change. The output is fully
  derived from the live constant graph; CI can run it under
  `--diff-only` mode to fail when the committed sidecars drift.
* The generator does **not** declare a return type for `#execute` —
  that one is yours to write.
