# 07 — Risks and Open Questions

Each entry has a **decision deadline** of "before tagging `v1.0.0`". Items
that affect the public surface must be resolved before
[`01-api-surface.md`](./01-api-surface.md) is marked **Frozen**.

## Open questions

### Q1. Should the result be a `Data` value object instead of a `Hash`?

- **Today** (`lib/assistant/service.rb:24`): `#run` returns a plain `Hash`.
- **Option A — keep the hash for 1.0 (current plan)**: zero churn for
  existing users; pattern matching still works
  (`run => { result:, status: }`).
- **Option B — return a `Data.define(:result, :status, :warnings, :errors)`**:
  immutable, self-documenting, but breaks `outcome[:result]` access for any
  caller using `Hash#[]`.
- **Recommendation**: A for 1.0; revisit for 2.0. Document under
  [`02-features.md`](./02-features.md) "Won't" with a 2.x marker.
- **Status**: `[x]` — **A** chosen. See resolution log.

### Q2. Should `valid_require_*?` be renamed to `valid_required_*?`?

- **Today** (`lib/assistant/input_builder.rb:48`): predicate is
  `valid_require_<name>?`.
- **Option A — leave as-is**: zero churn.
- **Option B — alias both, deprecate the old name in 1.x, remove in 2.0**:
  cleaner English; predictable cost.
- **Recommendation**: A for 1.0. The methods are predominantly internal —
  generated by the DSL and used by `validate_inputs`
  (`lib/assistant/service.rb:54`). Users rarely call them directly.
- **Decision**: **B** — alias `valid_required_*?` (new canonical name) to
  `valid_require_*?`; the deprecated name emits `Kernel.warn` once per call
  site and is removed in 2.0. Tracked as **M9** in
  [`02-features.md`](./02-features.md).
- **Status**: `[x]`.

### Q3. Should `if:` re-evaluate at execute time, or only at validate time?

- **Today**: only at validate time. The lambda receives the input value and
  decides whether the requirement is satisfied
  (`lib/assistant/input_builder.rb:60`).
- **Option A — keep validate-only semantics**: simplest; matches user
  expectation that validation is a one-shot phase.
- **Option B — re-run at execute boundary**: enables dynamic conditions
  based on results from a previous step, but couples validation and
  execution.
- **Recommendation**: A for 1.0.
- **Status**: `[x]` — **A** chosen.

### Q4. Should `inputs(...)` (plural) accept per-name overrides?

- **Today** (`lib/assistant/input_builder.rb:13`): every name in the array
  shares the same `type:` and options.
- **Option A — keep flat**: simple, predictable; users who want overrides
  call `input` individually.
- **Option B — accept a hash**: e.g.
  `inputs({a: { default: 1 }, b: { required: true }}, type: Integer)`.
  Powerful but adds API surface for a marginal gain.
- **Recommendation**: A for 1.0.
- **Status**: `[x]` — **A** chosen.

### Q5. Confirm the CI Ruby matrix entry `'4.0'`

- **Today** (`.github/workflows/ci.yml:21`): matrix is `['3.4', '4.0']`.
- **Risk**: if Ruby 4.0 has not shipped a stable release by the time we tag
  1.0.0, the matrix entry must be replaced (e.g. with `'3.5'`) — see
  [`05-quality-and-tooling.md`](./05-quality-and-tooling.md).
- **Recommendation**: re-evaluate within one week of cutting `v1.0.0.rc1`.
- **Decision**: Ruby 4.0 is stable; keep the floating `'4.0'` matrix entry
  (auto-tracks the latest 4.0.x via `ruby/setup-ruby`). Gemspec floor stays
  at `>= 3.4`.
- **Status**: `[x]`.

### Q6. YARD vs RBS as the canonical source of truth

- **Today**: neither is authoritative; YARD is not configured and `*.rbs`
  is empty.
- **Option A — RBS canonical, YARD as prose**: precise types, but RBS
  cannot express the per-input getters generated by the DSL precisely.
- **Option B — YARD canonical, RBS as best-effort**: more flexible prose,
  but no type-checker can validate it.
- **Recommendation**: B. Ship RBS as a best-effort artifact (Steep is an
  allowed-failure CI job per [`05-quality-and-tooling.md`](./05-quality-and-tooling.md)),
  document YARD as the source of truth.
- **Decision**: **A (reversed)** — RBS is canonical, YARD is supporting
  prose. Steep is a **required** CI job day one. To make this tractable
  given R1, we ship a bundled `bin/assistant-rbs` CLI that generates
  per-class `.rbs` for each `Assistant::Service` subclass; see **M11** in
  [`02-features.md`](./02-features.md).
- **Status**: `[x]`.

### Q7. Should `Service#status` consider info logs?

- **Today** (`lib/assistant/service.rb:47`): `status` is `:with_warnings`
  iff `warnings.any?`; info logs do not affect status.
- **Recommendation**: keep as-is. Info logs are explicitly informational and
  must not affect success/failure semantics.
- **Status**: `[x]` — confirmed.

### Q8. Should `LogItem` raise on invalid construction?

- **Today** (`lib/assistant/log_item.rb:10`): `LogItem.new(level: '', ...)`
  succeeds; `valid?` returns `false`. The library philosophy is soft-fail.
- **Recommendation**: keep soft-fail. A strict mode could be added in 1.x
  via an opt-in (`LogItem.strict!`) without breaking back-compat.
- **Decision**: **B (reversed)** — `LogItem.new` raises `ArgumentError`
  when any of `level`, `source`, `detail`, `message` is invalid. This is a
  **breaking change** from 0.1.0; existing `#valid?` / `#valid_level?` /
  `#valid_source?` / `#valid_detail?` / `#valid_message?` predicates are
  retained for introspection. Internal call sites
  (`log_item_error_initialize`, etc.) are audited so normal flows never
  trigger the raise. Tracked as **M10** in
  [`02-features.md`](./02-features.md) and documented as a breaking change
  in [`06-migration-0x-to-1.md`](./06-migration-0x-to-1.md).
- **Status**: `[x]`.

### Q9. Should we ship `examples/` inside the gem?

- **Recommendation**: no — keep `examples/` in the repo for documentation
  but exclude from `spec.files` in `assistant.gemspec`. Aligns with the
  existing exclusion pattern (`assistant.gemspec:31`).
- **Status**: `[x]` — confirmed; exclusion handled in
  [`04-release-checklist.md`](./04-release-checklist.md).

## Risks

### R1. Metaprogrammed methods limit RBS precision

The `Service.input` DSL generates per-name methods at class-definition time.
RBS has no way to express these without a per-class signature file. **Mitigation**:
ship a generic signature for `Service` and document the limitation in
`docs/guides/inputs.md` (per
[`03-documentation.md`](./03-documentation.md)).

### R2. SimpleCov gates may block legitimate refactors

A 95%-line / 90%-branch gate can be too strict for one-off refactors of the
metaprogrammed DSL. **Mitigation**: allow PR authors to lower the gate
temporarily with maintainer approval; never below 90/85.

### R3. RubyGems trusted publishing environment drift

`.github/workflows/release.yml:18` references `environment: rubygems`. If
the GitHub repository's environment configuration changes (e.g. on transfer
or rename), the release will fail silently from a contributor's POV.
**Mitigation**: add a manual `workflow_dispatch` dry-run step and document
the environment in `CONTRIBUTING.md`.

### R4. Ruby 4.0 may not ship in time

See Q5. **Mitigation**: the CI matrix is the only place that pins `'4.0'`;
swapping it is a one-line change.

### R5. Documentation rot between `01-api-surface.md` and YARD docs

Two sources of truth for the same surface always drift. **Mitigation**: in
the PR template (D4 in [`03-documentation.md`](./03-documentation.md)), add
a checkbox "Updated `docs/v1/01-api-surface.md` if the public surface
changed."
